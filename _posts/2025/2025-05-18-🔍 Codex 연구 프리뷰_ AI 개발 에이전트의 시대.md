# 🔍 Codex 연구 프리뷰: AI 개발 에이전트의 시대

2025년, 소프트웨어 개발은 이전과는 확연히 다른 풍경을 그리고 있습니다. 특히 '**바이브 코딩(Vibe Coding)**'이라는 흥미로운 개념이 개발자는 물론 비개발 직군에게까지 빠르게 확산되면서, 우리가 꿈꿔왔던 앱이나 웹사이트를 이전보다 훨씬 수월하게 만들 수 있는 시대가 열리고 있습니다.

> ❓ **바이브 코딩(Vibe Coding)이란?**  
> 출처: [위키백과, 바이브 코딩(Vibe Coding)이란?](https://ko.wikipedia.org/wiki/%EB%B0%94%EC%9D%B4%EB%B8%8C_%EC%BD%94%EB%94%A9)
>
> * **바이브 코딩(Vibe Coding)**은 2023년, 카파시(Karpathy)가 제시했던 "**가장 인기 있는 새로운 프로그래밍 언어는 영어**"라는 주장에서 한 걸음 더 나아간 개념입니다.
> * 이는 거대 언어 모델(LLM)의 놀라운 능력 덕분에, 더 이상 인간이 컴퓨터에게 명령하기 위해 복잡한 프로그래밍 언어를 애써 배우지 않아도 된다는 혁신적인 변화를 의미합니다.

OpenAI가 공개한 **Codex**는 단순한 코드 생성기가 아니라, 실제 개발 업무를 자동화하고 협업할 수 있는 **AI 소프트웨어 엔지니어링 에이전트**입니다.

![](https://velog.velcdn.com/images/euisuk-chung/post/f213a6eb-132a-4584-9dcb-9bcff5f72786/image.png)

이 글에서는 Codex가 어떤 원리로 작동하는지, 어떤 작업을 자동화할 수 있는지, 그리고 발표 영상의 실제 데모 사례를 바탕으로 Codex의 현재와 미래를 소개합니다.

💡 Codex란?
---------

**Codex**는 `OpenAI 클라우드 기반에서 작동`하는 **AI 개발 에이전트**입니다.

단순한 코드 자동 완성 수준을 넘어, 다음과 같은 기능을 수행합니다:

* GitHub 리포지토리를 로드하고 코드베이스를 분석
* 버그 탐지 및 수정, 테스트 코드 생성 및 실행
* PR 작성 및 커밋 메시지 생성
* 코드 스타일 및 린터 규칙에 맞게 포맷팅
* 여러 작업을 병렬로 처리하며, 각각을 독립된 VM에서 안전하게 실행

🔧 Codex는 **GPT 기반 모델인 `codex-1(코덱스-1)`**을 기반으로 하며, 이는 실제 프로덕션 환경에서 병합 가능한 수준의 코드 품질을 목표로 학습되었습니다.

* 코덱스-1의 훈련 목표은 "*결과물을 사람의 코딩 선호도 및 표준에 맞춰 조정하는 것*"이라고 OpenAI는 밝혔습니다.
* 이는 o3와 비교했을 때, 사람이 **즉시 표준 워크플로우에 통합할 수 있을 정도로 더 깔끔한 패치를 생성**한다고 설명했습니다.

> **사용 방법**

💻 ChatGPT의 사이드바를 통해 Codex에 접속하여 프롬프트를 입력하고 사용할 수 있습니다.

* "**Code**"를 클릭하여 새로운 코딩 작업을 할당할 수 있습니다.

코드베이스에 대해 Codex에 `질문하고 싶다면` "**Ask**"를 클릭하세요.

* 각 작업은 코드베이스가 미리 로드된 **격리된 독립 환경**에서 개별적으로 처리됩니다.
* 작업 소요시간은 **1분에서 30분** 정도로, 복잡도에 따라 달라집니다.
* Codex는 코드 파일을 **읽고 수정**할 수 있으며, **명령어 실행**(예: 테스트 실행, 린터, 타입 검사 등)도 가능합니다.
* 작업이 수행되는 동안 Codex의 진행 상황을 **실시간으로 모니터링**할 수 있습니다.

> (참고) **코드 베이스란?**

* **하나의 소프트웨어 프로젝트 전체에 해당하는 소스 코드의 집합**을 의미합니다.
  + 보통 하나의 프로젝트는 여러 개의 파일과 디렉토리로 구성되어 있으며,  
    이 모든 코드를 통틀어 "하나의 코드베이스"라고 부릅니다.
  + 이 코드에는 **비즈니스 로직**, **API 정의**, **데이터 처리 코드**, **UI 구성 요소**, **설정 파일**, **테스트 코드** 등이 포함됩니다.

작업 완료 후 `Codex는 자체 환경 내에서 변경 사항`을 **커밋(commit)**합니다.

* Codex는 **터미널 로그**, **테스트 출력 결과** 등 명확한 증거를 통해 자신이 수행한 작업을 **검증 가능한 방식**으로 제공합니다.
* 사용자는 Codex의 결과를 **리뷰**, **재수정 요청**, **GitHub Pull Request 생성**, 또는 **로컬 환경으로 직접 통합**할 수 있습니다.
* 또한, 실제 개발 환경과 최대한 유사하도록 Codex 환경을 **사용자 설정(configure)**할 수 있습니다.

### 📘 Codex를 안내하는 파일: `AGENTS.md`

Codex는 사람처럼 코드를 읽고 작업하지만, **문맥과 규칙을 더 잘 이해하도록 안내할 수 있는 구성 파일**이 존재합니다.  
바로 `AGENTS.md`입니다.

> 🤖 **AGENTS.md는 "AI 개발자를 위한 가이드라인 문서"**입니다.
>
> * **인간 개발자**가 프로젝트에 처음 합류했을 때 `README.md`, `CONTRIBUTING.md`, `docs/` 등을 읽으며 코드를 이해하듯,
> * **Codex**는 `AGENTS.md`를 읽고 **“이 프로젝트는 이렇게 이해하고, 이렇게 작업하면 돼요”**라는 **지침(instruction)**을 받습니다.

| 항목 | `README.md` | `AGENTS.md` |
| --- | --- | --- |
| 🎯 **대상** | 인간 개발자 (사람) | Codex 같은 AI 개발 에이전트 |
| 📚 **목적** | 프로젝트 소개, 설치법, 사용법 안내 | 코드 탐색 방식, 테스트 실행 방식, 스타일 가이드 **자동 이해용 지침** |
| 📄 **형식 자유도** | 문장 위주, Markdown 스타일 자유로움 | **구조화된 지침 위주**, 실행 가능한 명령이나 규칙 전달 |
| ⚙️ **실행 기반** | 사람이 보고 직접 실행 | Codex가 **자동 분석, 실행, 테스트**하는 데 참고 |

**🔧 `AGENTS.md`는 무엇을 하나요?**

Codex는 프로젝트 내에 `AGENTS.md` 파일이 존재하면, 다음과 같은 방식으로 이를 참고하여 **작업 정확도를 높입니다**:

| 역할 | 설명 |
| --- | --- |
| 🧭 코드 구조 안내 | 폴더/모듈 간의 관계나 진입점을 Codex에게 설명합니다. |
| 🧪 테스트 명령 지정 | 예: `pytest`, `pnpm test` 등 Codex가 자동으로 테스트를 실행할 수 있도록 지시 |
| 🧑‍💻 코드 스타일 및 규칙 안내 | 예: `"Black으로 포맷팅"`, `"커밋 메시지는 Conventional Commits 사용"` 등 |
| 📁 리포지토리 탐색 힌트 | 어떤 디렉토리를 무시해야 하는지, 무엇을 먼저 읽어야 하는지 등 명시 가능 |

---

**📂 예시 구성 (`AGENTS.md`)**

```
  # AGENTS.md

  ## Code Style
  - Use Black formatter for all Python files.
  - Follow PEP8 for naming conventions.

  ## Testing
  - Run `pytest tests/` before submitting a PR.
  - Ensure test coverage does not drop.

  ## Repository Structure
  - `src/` contains main application code.
  - `scripts/` contains developer utilities.

  ## PR Guidelines
  - PR title format: `[Fix] Short description`
  - Include test results in the description.
```

> (참고) Codex는 `AGENTS.md` 없이도 작동하지만, 이 파일이 있을 경우 **훨씬 더 높은 품질의 결과를 생성**합니다.

🧪 발표 영상으로 본 Codex 사용 시나리오
-------------------------

OpenAI는 Codex의 기능을 시연하는 실제 데모를 통해 다음과 같은 사용 흐름을 소개했습니다.

> 참고로 현재 Plus User들에게는 오픈되지 않았으므로, Demo 영상 내용을 다룹니다.  
> (Images are from OpenAI Youtube Demo Talk)

### 1. GitHub 연동 및 리포지토리 선택

* 사용자는 GitHub 계정을 Codex에 연동하고, 원하는 리포를 선택합니다.
* 예시로 사용된 리포: `preparedness`, `codex-cli`
* Codex는 리포지토리를 분석하고, 코드 구조를 파악합니다.

### 2. 다양한 작업(Task) 병렬 실행

사용자는 여러 작업을 동시에 할당할 수 있습니다:

| 작업 종류 | 설명 |
| --- | --- |
| Ask | 코드베이스의 구조나 특정 함수에 대한 설명 요청 |
| Code | 코드 내 버그를 찾아 수정 |
| Suggest | 코드 개선 아이디어 제안 |
| Fix Typos | 오탈자 및 문법 오류 자동 수정 |

### 3. 실제 작업 수행 예시

다음은 데모에서 수행된 실제 작업입니다:

🖥️ **DEMO 1.** 코드 유지보수성 및 버그 가능성 검토(Maintainability & Bugs)

사용자가 Codex에게 다음과 같은 작업을 요청합니다:

![](https://velog.velcdn.com/images/euisuk-chung/post/3bec37f4-3e5d-4ce2-b790-724828a4b939/image.png)

> *“이 코드베이스를 유지보수 가능하고 버그가 없도록 유지하고 싶습니다. TODO 외의 부분에서 개선이 필요한 부분을 읽고, 그 이유와 함께 작업 제안(task)을 해주세요.”*

Codex는 코드베이스를 분석한 후, **신뢰성과 유지보수성을 향상시킬 수 있는 영역**을 다음과 같이 식별합니다:

✅ **1. Avoid mutable default arguments**

![](https://velog.velcdn.com/images/euisuk-chung/post/c1641ce7-eb28-4118-bbe0-9c709401029f/image.png)

* `mutable default argument` 문제를 감지하고 함수 시그니처 수정

  + 파이썬에서 함수의 기본 인자는 함수가 호출될 때가 아니라, 정의될 때 딱 한 번만 만들어지기 때문에
  + **기본 인자로 전달된 {}**는 함수 바깥에서 한 번 만들어져서 계속 공유된다는 게 문제입니다.

  ![](https://velog.velcdn.com/images/euisuk-chung/post/242eccef-a0e2-4477-a914-16e49841d83c/image.png)

  > 각각의 task들이 개별 SANDBOX에서 실행되고 있다는 예시

  (참고) ⌛ **샌드박스(Sandbox)**는 외부 시스템으로부터 격리된(분리된) 환경을 의미하며, 코드 실행, 실험, 분석, 테스트 등을 안전하게 수행하기 위해 사용됩니다.

✅ **2. Correct variable spelling**

![](https://velog.velcdn.com/images/euisuk-chung/post/846cabef-d144-45c5-80a7-a7c548408a78/image.png)

* 오탈자 및 변수 오기 정정

✅ **3. Consistent timeout constant**

![](https://velog.velcdn.com/images/euisuk-chung/post/0b4d4a29-4c6b-418b-99fd-6550e2753957/image.png)

* timeout 설정 값의 불일치(예: 60초 vs 120초)를 탐지하고 일관성 있게 수정

✅ **4. Ensure recorder writes are flushed**

![](https://velog.velcdn.com/images/euisuk-chung/post/3bee7bc5-f71b-4853-9841-198b6f3dae48/image.png)

* 사용자의 목표("코드베이스를 유지 보수하기 쉽게")에 맞는 개선 제안

🖥️ **DEMO 2.** 특수문자 파일명에서 `/diff` 명령 오류 해결 요청 사례

사용자가 Codex에게 다음과 같은 CLI 버그 수정을 요청합니다:

![](https://velog.velcdn.com/images/euisuk-chung/post/fd8c1077-df88-44c9-a6c9-7ea1f5950853/image.png)

> *“`/diff` 명령어가 특수문자(`$`)가 포함된 파일명을 처리하지 못해 오류가 발생합니다. 이를 해결해주세요.”*
>
> * 예: `'a$b.txt'` 파일을 만든 뒤 `/diff` 실행 시 에러 발생

---

Codex는 CLI 환경에서 문제를 재현하기 위해 **샌드박스(격리된 컨테이너 환경)**를 시작하고, 아래의 과정을 통해 문제를 분석합니다:

✅ **1. 재현 스크립트 자동 생성 및 실행 시도**

![](https://velog.velcdn.com/images/euisuk-chung/post/36a0e545-359c-4adf-b90b-2639bd19eba3/image.png)

* `/tmp/repro.ts`에 TypeScript 스크립트를 생성

  + 임시 Git 저장소 생성
  + 특수문자 `$`가 포함된 파일 `'a$b.txt'`를 추가
  + 내부 diff 함수를 호출하여 오류 발생 여부 확인

![](https://velog.velcdn.com/images/euisuk-chung/post/1dc82533-060e-478c-ade6-b3e3c990f625/image.png)

* 실행 결과 `get-diff.js` 모듈 경로 문제 발생 → Codex가 원인 파악 후 다음 단계로 진행

---

✅ **2. 오류 수정 및 모듈 패치**

![](https://velog.velcdn.com/images/euisuk-chung/post/dc901926-de53-4c69-a69f-65c93e1e6d86/image.png)

* 핵심 수정: `child_process`에서 `execFileSync`를 명시적으로 사용하여 **셸을 우회**

  + 기존 `execSync()`는 `$`를 셸 변수로 오인함
  + `execFileSync()`는 문자열을 그대로 전달하므로 **특수문자 안전**
* 변경사항 적용 후 regression test까지 작성 완료

---

✅ **3. 테스트 자동화 및 검증 완료**

![](https://velog.velcdn.com/images/euisuk-chung/post/23f6e8e2-1ace-4f81-be5a-de2612a91efe/image.png)

* `pnpm test`로 새로운 테스트 통과 확인
* `$` 문자가 포함된 파일도 diff 결과에 정상적으로 포함되는 것을 확인

---

✅ **4. Git 로그 정리 및 PR 준비 완료**

![](https://velog.velcdn.com/images/euisuk-chung/post/f95ba8e8-98bc-4416-b7e8-900fd1a7ab19/image.png)

* Codex는 수정 내용을 커밋하고 요약:

  + `get-diff.ts`: 기능 수정
  + `get-diff-special-chars.test.ts`: 회귀 테스트 추가
* 커밋 메시지:

  > *"fix diff for special chars"*  
  > 2 files changed, 40 insertions(+), 6 deletions(-)

🧠 Codex는 자신이 제안한 작업을 **스스로 실행하고, 테스트하고, PR까지 생성**할 수 있습니다.

### 4. 테스트 및 결과 검토

Codex는 작업을 완료한 후 다음 정보를 제공합니다:

🛡️ Codex의 안전성과 신뢰성
------------------

Codex는 다음과 같은 방식으로 **투명성과 안전성**을 확보합니다:

* **격리된 VM 환경**에서만 작동 (인터넷 차단)
* 모든 작업의 **터미널 로그, 테스트 결과, 코드 인용 기록** 제공
* **AGENTS.md** 파일을 통해 테스트 방법, 커밋 스타일, PR 포맷 등 세부 사항을 학습
* 테스트 실패나 의심되는 코드에 대해 **명확하게 경고**하며, 수동 리뷰를 권장

🎯 Codex가 지향하는 개발 문화
-------------------

Codex는 단순히 자동화를 넘어서, **개발자의 생산성을 극대화**하는 도구입니다:

* **자동화**: 반복 작업(리팩토링, 포맷팅, 테스트 작성 등)을 위임
* **병렬 처리**: 여러 에이전트를 동시에 활용하여 다중 작업 병렬 실행
* **비동기 협업**: 점심시간에 작업을 시작하고, 돌아와서 결과를 확인할 수 있음
* **소프트웨어 공학 강화**: 테스트 환경, 문서화, 구조 정리가 잘 된 프로젝트에서 성능 극대화
